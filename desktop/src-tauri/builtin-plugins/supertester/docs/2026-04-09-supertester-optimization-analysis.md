# Supertester 反向优化分析

基于对比报告 [2026-04-08-comparison.md](E:/workspace/aise/TestingAgent/docs/2026-04-08-comparison.md) 与当前 skill 设计的交叉分析，本文聚焦三个问题：

1. Supertester 当前生成结果暴露了哪些稳定问题
2. 这些问题背后有哪些通用特性模式
3. 应该如何反向修改 agent 设计，优先补齐哪些能力

## 1. 结论摘要

对比结果不是简单的“AI 不如人工”，而是两套测试设计机制在优化目标上不同：

- Supertester 优先优化“结构化覆盖”和“可枚举覆盖”
- 人工测试设计优先优化“业务高保真”和“真实交互细节”

这使得 Supertester 在以下维度表现稳定领先：

- 模块覆盖完整性
- 跨模块场景
- 隐含需求挖掘
- 安全与异常覆盖
- 枚举/边界/矩阵式测试

也使它在以下维度稳定落后：

- 文案、图片、链接、视觉等高保真 UI 资产验证
- loading、刷新、切语言、退出登录等中断型 UX 场景
- prompt 模板、AI 合约、日志路径等“内容即需求”的验证
- 历史记录排序、分页、滚动等细碎交互
- PRD 之外但真实存在的运营/配置型业务知识

这说明当前 Supertester 的主问题，不是“不会生成测试”，而是“对某些需求类型没有建立足够强的触发与守门机制”。

## 2. 真实差异中暴露出的核心问题

### 2.1 偏向可枚举规则，弱化高保真内容资产

对比报告显示，Supertester 很擅长把黑名单域名、边界值、状态矩阵、安全规则拆成独立用例，但在下面这些场景里持续漏测：

- loading 五阶段文案
- 首页宣传模块的标题、描述、图片
- Logo 不变形、版权链接跳转
- hover 文案
- prompt 模板逐字段验证
- AI 诊断日志路径

这类需求的共同点是：

- 需求对象不是“动作”而是“内容本身”
- 价值不在流程是否完成，而在展示/输出是否精确
- 往往要求逐项、逐字段、逐状态校验

当前技能虽然在 `requirement-analysis` 中已经提出“测试资产”概念，也列出了内容资产、规则资产、约束资产，但从生成结果看，这个抽取规则还没有稳定传导到 Phase 3 的用例粒度决策。

### 2.2 偏向主流程与系统视角，弱化用户中断与恢复体验

人工用例明显强于 Supertester 的一个区域，是“用户行为被打断后系统怎么继续工作”。典型遗漏包括：

- 页面刷新后的恢复
- 检测过程中的语言切换
- 生成中的菜单切换/退出登录
- loading 过程的阶段反馈
- 列表排序、分页、滚动的连续体验

这说明 Supertester 当前对状态机的理解仍然偏后端式：

- 更容易覆盖“系统状态 A 到 B”
- 不容易主动生成“用户打断中间态”场景

也就是说，它对“最终状态”敏感，对“过程反馈”和“中断恢复”不够敏感。

### 2.3 对“内容即合约”的识别仍然偏弱

在 AI 可见性分析、GEO 内容创作等模块，人工用例对 prompt 模板逐字段验证、系统提示词校验、AI 输出契约校验都明显更细。这类需求的本质不是 UI，也不是普通业务规则，而是：

- 模板文本即业务逻辑
- 输出 schema 即业务合约
- 文案/字段顺序/拼装规则即系统正确性的组成部分

Supertester 目前已经有“约束与合约资产”的概念，但实际结果说明：

- 它能识别 schema 型合约
- 但不擅长识别 prompt/content template 也是合约

这会导致它把“验证 prompt 内容”弱化成“验证结果大致正确”。

### 2.4 缺少对视觉/媒体资产的专门降级策略

报告里几个典型缺口是：

- 宣传模块图片验证
- Logo 不变形
- 视觉展示类细节

这类能力并不一定要求 agent 真正具备视觉理解，而是至少需要在生成阶段做两件事：

1. 明确识别这类需求属于“视觉资产”
2. 自动降级成 manual 或 partial case，而不是直接遗漏

当前 Supertester 的问题不是只“不会测图”，而是“没有把不会测图这件事稳定记录成待人工验证的资产”。

### 2.5 对 PRD 外业务知识几乎无感知

报告中“早鸟限免模式”是很典型的例子。人工测试能覆盖，是因为写用例的人知道：

- 真实业务不只存在于 PRD
- 还存在于运营配置、历史版本、默认开关、旧行为兼容中

Supertester 虽然在 `using-supertester` 里已经强调“已有测试资产不能只当参考材料”，但它当前仍主要依赖明确输入的需求文本与上下文文件。结果就是：

- 对 PRD 外真实业务盲区较大
- 对“历史遗留/运营配置/下线功能”类需求识别不足

这不是单纯的生成策略问题，而是输入域边界问题。

### 2.6 test-reviewer 审查维度还不够贴近人工 QA 的质检方式

`test-reviewer` 现在主要审：

- 需求覆盖
- 前置条件
- 步骤清晰度
- 预期结果可验证性
- 去重与可追溯性

它缺少一组非常关键的“高保真覆盖质检项”：

- 是否遗漏逐字段文案校验
- 是否遗漏中断恢复类场景
- 是否遗漏排序/分页/滚动等列表行为
- 是否遗漏视觉资产的 manual 标记
- 是否把 prompt/template 当作合约验证

换句话说，当前 reviewer 更像“结构正确性审查”，还不够像“资深测试设计师的覆盖盲区审查”。

## 3. 可提炼的通用特性模式

从这次真实需求对比里，可以反推出几类“Supertester 必须显式建模”的需求特性。

### 3.1 高保真内容资产

特征：

- 文案、提示词、模板、邮件、hover、空状态、loading 文案
- 验证目标是“内容本身”而不是“功能可用”
- 常见关键词：文案、提示、模板、内容如下、字段如下、显示为、按钮文字、提示语

优化含义：

- 不能只抽取为内容资产
- 必须驱动用例粒度变细
- 必须驱动 reviewer 检查“是否逐项验证”

### 3.2 可视反馈与过程性状态

特征：

- loading 阶段
- 进度条/状态提示
- 扫描/生成/分析中的过程切换
- 成功前的中间反馈

优化含义：

- 不能只验证最终成功态
- 必须显式生成“过程态验证用例”

### 3.3 中断恢复型交互

特征：

- 刷新
- 切语言
- 页面跳转再返回
- 菜单切换
- 重新登录/退出登录
- 网络中断后恢复

优化含义：

- 需要作为单独生成触发器，而不是寄希望于状态机场景顺带覆盖

### 3.4 视觉/媒体资产

特征：

- 图片
- Logo
- 视觉样式
- 排版与布局相关断言

优化含义：

- 需要单独识别
- 自动转为 `manual` 或 `partial`
- 在报告中明确列为“保留人工验证资产”

### 3.5 内容模板/Prompt/Schema 合约

特征：

- AI prompt
- 输出模板
- JSON schema
- 多字段拼装规则
- 日志路径或文件路径格式

优化含义：

- 视为“合约资产”
- 必须逐字段、逐结构验证
- 不能只用黑盒结果覆盖

### 3.6 运营配置与 PRD 外业务知识

特征：

- 早鸟限免
- 下线开关
- 灰度规则
- 历史兼容行为
- 后台配置影响前台逻辑

优化含义：

- 需要在 Phase 1 明确询问“是否存在 PRD 外业务规则/运营开关/历史兼容”
- 需要允许外部测试资产提升优先级

## 4. 这些问题是怎样在当前 agent 里形成的

### 4.1 requirement-analysis 已经定义了资产类型，但触发词仍偏弱

`requirement-analysis` 已经明确要求提取：

- 内容资产
- 规则/枚举资产
- 集成资产
- 状态/数据资产
- 合约资产

但对真实需求而言，当前问题不是“完全没定义”，而是：

- 缺少高频触发词到资产类型的强映射
- 缺少“发现这类资产后必须提升用例粒度”的强制规则

结果是资产被识别了，但后续并没有稳定转换成更细的测试设计。

### 4.2 test-case-generation 对“生成器选择”强调很多，但对“保真策略选择”强调不够

当前 Skill 3 已强调：

- 按需求类型选生成器
- 保留关键测试资产
- 避免去重吞掉高价值资产

但它还缺一层更明确的策略分流：

- 什么情况下要拆成逐字段校验
- 什么情况下要生成中断恢复用例
- 什么情况下要输出 manual/partial 的视觉校验
- 什么情况下要对历史记录生成排序/分页/滚动用例

也就是说，当前生成策略偏“选哪种生成器”，而不够偏“用什么保真粒度生成”。

### 4.3 requirement-association 擅长跨模块，却不擅长“跨状态中断”

Skill 2 对以下问题已经很强：

- 关键路径
- 错误传播
- 并发
- 数据同步
- 证据链
- 共享资源

但“用户中断恢复”不是纯粹的模块关联问题，也不是普通状态迁移问题，它更像：

- 半路打断
- 页面重建
- UI 状态丢失/恢复
- 过程性提示是否继续一致

这类场景需要单独的中断模板，否则很难自然冒出来。

### 4.4 reviewer 缺少“高保真缺口雷达”

当前 reviewer 可以判断覆盖漏项，但还没有明确检查以下缺口：

- 需求中出现“内容如下/模板如下”却没有逐项验证
- 需求中出现 loading/进度/处理中，却只测最终态
- 需求中出现历史列表，却没测排序/分页/滚动
- 需求中出现图片/Logo，却没转成 manual/partial
- 需求中出现 prompt/schema/log path，却未作合约校验

因此很多结果虽然“结构合理”，但离人工高质量用例仍差一步。

## 5. 对 Supertester 的反向优化建议

## 5.1 P0：补“需求特性识别”而不是先补更多生成器

优先级最高的不是继续增加生成器数量，而是增强特性识别和策略触发。

建议在 `requirement-analysis` 增加 6 类强触发标签：

- `content_fidelity`
- `process_feedback`
- `interruption_recovery`
- `visual_asset`
- `contract_content`
- `business_outside_prd`

每个 F-xxx 在解析时都额外标注这些标签，供后续阶段使用。

### 5.2 P0：在 Phase 1 增加“PRD 外资产”澄清问题

建议在 `requirement-analysis` 中增加一条固定澄清分支：

- 是否存在运营开关、历史兼容、后台配置、已下线但需验证的旧行为？

如果用户回答有，必须写入 `parsed-requirements.md` 的资产区，并提升为后续用例生成输入。

这能直接补“早鸟限免”类盲区。

### 5.3 P0：给 test-case-generation 增加“保真度策略”

在 Skill 3 的 Step 1 后增加一层决策：

- `enumeration mode`：完整枚举/矩阵
- `content fidelity mode`：逐字段/逐文案校验
- `process mode`：过程态与 loading 校验
- `interruption mode`：刷新/切换/恢复
- `visual fallback mode`：manual/partial
- `history interaction mode`：排序/分页/滚动

这层策略不是替代子生成器，而是决定最终用例粒度与类型。

### 5.4 P0：把 prompt/template 明确升级为一级资产

建议在 `requirement-analysis` 和 `test-case-generation` 中把以下内容单列：

- prompt 模板
- AI 系统提示词
- 输出 schema
- 拼装模板
- 日志路径/导出路径

并要求：

- 出现这些资产时，默认生成至少一个“结构校验”用例
- 如内容复杂，再生成逐字段校验用例

### 5.5 P0：强化 reviewer 的缺口审查清单

建议给 `test-reviewer` 新增专门检查项：

- 是否遗漏高保真内容校验
- 是否遗漏过程态/loading 校验
- 是否遗漏中断恢复场景
- 是否遗漏历史列表交互校验
- 是否遗漏视觉资产的 manual/partial 标记
- 是否将 prompt/template 作为合约校验
- 是否遗漏 PRD 外业务资产

并将其中几项设为 HIGH：

- 明确出现的内容模板未逐项验证
- 明确出现的 loading/process 只测最终态
- 明确出现的图片/Logo 未标记 manual/partial

### 5.6 P1：新增“中断恢复场景模板”

建议在 `requirement-association` 或 `test-case-generation` 里增加一个固定模板组：

- refresh during processing
- switch language during processing
- navigate away and return
- logout/login during pending state
- retry after network recovery

这组模板不依赖具体业务领域，适合作为通用生成规则。

### 5.7 P1：新增“列表交互完整性模板”

对所有存在历史记录、表格、列表、记录流的模块，默认检查是否需要：

- 排序
- 分页
- 滚动加载
- 状态筛选
- 空状态

这类场景在人工用例里出现频率很高，但当前 AI 输出中明显偏弱。

### 5.8 P1：把视觉缺口系统化输出到最终报告

对于视觉类、媒体类、品牌类资产，即使无法自动化验证，也应在最终报告中明确输出：

- 哪些是已覆盖行为
- 哪些需要人工视觉验收
- 哪些图片/Logo/样式资产未自动化

这样可以把“不会自动化”变成“明确保留的人工资产”，而不是静默漏掉。

## 6. 可以直接修改到 skill 的规则清单

### 6.1 修改 `skills/requirement-analysis/SKILL.md`

建议补充：

- 固定识别标签体系：高保真内容、过程态、中断恢复、视觉资产、内容合约、PRD 外业务资产
- 固定澄清问题：是否存在运营配置、历史规则、灰度开关、已下线但仍需回归的行为
- 固定触发语义：文案如下、模板如下、提示如下、图片如下、日志路径如下、历史记录、分页、排序、loading、处理中、切换语言、刷新后

### 6.2 修改 `skills/test-case-generation/SKILL.md`

建议补充：

- 生成器之外的“保真度模式”
- 对内容模板默认生成结构校验
- 对 loading/process 默认生成过程态用例
- 对图片/Logo 默认生成 manual/partial case
- 对历史列表默认检查排序/分页/滚动

### 6.3 修改 `skills/requirement-association/SKILL.md`

建议补充：

- 中断恢复型关联场景
- 页面重建后的状态恢复
- 进行中任务被切换上下文后的可恢复性

### 6.4 修改 `agents/test-reviewer.md`

建议补充：

- 高保真覆盖缺口审查维度
- 视觉资产降级审查
- 内容合约审查
- 历史列表与过程态覆盖审查

## 7. 推荐的实施顺序

### 第一阶段

- 修改 `requirement-analysis`
- 修改 `test-case-generation`
- 修改 `test-reviewer`

目标：

- 不改变整体 6 阶段结构
- 先补识别、触发、审查三处最核心的机制

### 第二阶段

- 修改 `requirement-association`
- 增加中断恢复模板
- 增加历史列表模板

目标：

- 让“真实交互型缺口”稳定浮出水面

### 第三阶段

- 调整 `test-reporting`
- 输出“人工保留资产”和“高保真未自动化资产”

目标：

- 让最终报告更贴近资深 QA 的可交付结果

## 8. 最关键的一条原则

这次对比最值得沉淀的一点是：

Supertester 不应该只追求“更多测试用例”，而应该追求“覆盖结构 + 资产保真”同时成立。

如果继续只强化结构化枚举能力，它会越来越像一个优秀的“系统性用例扩写器”。
如果把高保真内容、过程态、中断恢复、视觉降级、PRD 外业务资产也纳入强约束，它才会更接近一个真正可用的测试设计 agent。
