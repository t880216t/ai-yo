# 模糊需求识别模式参考

## 语言信号

### 模糊量词
- "等"、"之类"、"相关"、"相应"
- "合适的"、"合理的"、"适当的"
- "一些"、"若干"、"多个"
- "足够的"、"必要的"

### 缺少具体数值
- "在合理时间内" -> 多少秒/分钟？
- "密码需要足够安全" -> 最小长度？必须包含哪些字符？
- "支持大量用户" -> 并发上限是多少？
- "及时响应" -> SLA 是多少？

### 隐含假设
- "用户可以登录" -> 哪些登录方式？邮箱？手机？SSO？
- "显示错误信息" -> 什么格式？本地化？
- "重定向到首页" -> 什么条件下？所有角色都一样吗？

### 缺少异常处理
- 没有描述失败情况
- 没有描述边界情况
- 没有描述并发情况
- 没有描述网络异常

### 业务规则不完整
- "满足条件时" -> 具体条件是什么？
- "根据角色显示" -> 有哪些角色？权限矩阵？
- "按优先级排序" -> 优先级如何定义？相同优先级怎么排？

## 检测流程

```
对每个功能 (F-xxx):
  1. 检查是否有具体的验收标准（不只是"应该可以"）
  2. 检查数值是否明确（不是"合理"或"适当"）
  3. 检查是否描述了错误/异常情况
  4. 检查是否描述了边界条件
  5. 检查前置条件是否明确
  6. 检查后置条件是否明确
  7. 检查是否有多种理解方式
```

## 澄清问题生成原则

1. **具体化** — 不问"这个需求清楚吗？"，而是"最大登录尝试次数是多少？"
2. **提供选项** — "密码最小长度：A) 6位 B) 8位 C) 12位 D) 其他"
3. **一次一问** — 不在一条消息中问多个问题
4. **关联上下文** — 说明为什么这个信息对测试很重要
5. **优先级排序** — 先问影响测试覆盖范围的关键问题

## 常见领域模式

### 用户认证
- 支持哪些登录方式？
- 密码策略（长度、复杂度、过期）？
- 登录失败锁定策略？
- Session/Token 过期时间？
- 多设备登录策略？

### 电商/支付
- 支持哪些支付方式？
- 退款流程和时限？
- 库存并发扣减策略？
- 优惠券叠加规则？

### 表单/数据
- 字段长度限制？
- 必填/可选？
- 格式验证规则？
- 字符集支持（中文、emoji）？

### API/接口
- 速率限制？
- 认证方式？
- 错误响应格式？
- 分页策略？
